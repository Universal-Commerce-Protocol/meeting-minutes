## 2. Technical Discussions & Decisions

- **Topic:** Consent Submission
  - **Discussion:** Debated whether buyer marketing consent should be submitted during the update call or only upon the transaction complete call. Members raised concerns about user expectations, noting that users generally do not expect their contact data to be shared for marketing until they finalize a purchase. It was noted that collect-on-update is sometimes used for abandoned cart recovery, but for primary transaction logic, collecting on complete is more aligned with user expectations.
  - **Decision:** Aligned that buyer consent submission is restricted to the 'complete' transaction call, provided that the consent action has no side effects on the underlying transaction logic or pricing calculations.
  - **Context:** [#407 feat: add marketing consent extension for per-channel opt-in](https://github.com/Universal-Commerce-Protocol/ucp/pull/407)

## 3. Open Discussions & TC PR Reviews

- **Topic:** Fulfillment and Omni-Channel Retail Locations
  - **Notes:** Reviewed a proposal regarding retail locations for fulfillment. Feedback highlighted concerns that the initial design was too narrowly focused on retail pickup, and recommended a slight data model adjustment to treat lockers and in-store pickup as peers in an omni-channel solution. A technical deep dive will be scheduled for next week.

- **Topic:** Buyer Consent Extension V2 Design
  - **Notes:** Ilya Grigorik presented an alternative design for buyer consent that moves away from high-level booleans to fine-grained segments using a reverse DNS structure (e.g., `consent.email`, `consent.sms`). This allows custom segment-level consent while keeping standard compliance-oriented categories.
