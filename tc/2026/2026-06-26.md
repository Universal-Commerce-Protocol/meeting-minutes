## 2. Technical Decisions & Consensus

- **Topic:** Domain Working Group for Bulk Product Discovery
  - **Discussion:** The council debated whether to proceed directly with PRs or establish a dedicated group to discuss complex design aspects of the Bulk Product Discovery RFC, such as feed formats, diagnostic feedback, and row-level data control.
  - **Agreement:** Establish a domain working group (DWG) to deliberate on these design aspects before bringing it back to the tech council.
- **Topic:** Saved Payment Methods Extension
  - **Discussion:** Evaluated a proposal to create a separate extension for saved payment methods and addresses.
  - **Agreement:** Avoid creating a separate extension, choosing instead to leverage the existing identity linking infrastructure and documentation to support saved payment methods.
  - **Context:** [PR #519 Saved payment instruments w/ identity linking](https://github.com/Universal-Commerce-Protocol/ucp/pull/519)
- **Topic:** Closure of PR 288
  - **Discussion:** Reviewed outstanding payment constraints PRs.
  - **Agreement:** Agreed to close Pull Request 288, as the associated discussion has migrated to Pull Request 424.
  - **Context:** [PR #288](https://github.com/Universal-Commerce-Protocol/ucp/pull/288) / [PR #424](https://github.com/Universal-Commerce-Protocol/ucp/pull/424)

## 3. Open Discussions & TC PR Reviews

- **Topic:** Proposal for Bulk Product Discovery (RFC #550)
  - **Notes:** James Andersen presented a proposal to standardize product catalog data interchange in UCP, addressing the scalability limitations of real-time search/lookup APIs. Currently, platforms rely on web crawling or custom integrations. Amit Handa raised concerns regarding migration costs and confusion relative to existing Meta and Google ad-tech feeds. Vinod Kumar explained that UCP focuses on public, non-sensitive product data, whereas existing ad-tech integrations handle sensitive pricing and inventory details, suggesting parallel integrations initially. Rob Cooper noted the intent is to offer businesses a suite of format options. Standardized columns for data usage (e.g., controlling use for AI model training) were also discussed. Aaron Glazer and Vic Duggan expressed strong support, noting similar merchant demands for standardization.
  - **Context:** [#550: Feat: A Bulk Product Discovery Layer for UCP](https://github.com/Universal-Commerce-Protocol/ucp/pull/550)
- **Topic:** RFC Approval Process for Repository Refactoring
  - **Notes:** Jing Li requested that the council review and provide feedback on the current Repository Refactoring RFC (#520) to facilitate a formal vote. Amit Handa asked participants to document any strong objections on the RFC; otherwise, the first phase of changes will proceed.
  - **Context:** [#520: Proposal: Repository Refactoring for Multi-Vertical Support](https://github.com/Universal-Commerce-Protocol/ucp/pull/520)
- **Topic:** PR 519 (Saved Payment Methods with Identity Linking)
  - **Notes:** James Andersen introduced PR 519, which leverages identity linking to support saved payment methods and addresses. Daniel Wyckoff and Amit Handa supported this, agreeing that documenting how to use existing infrastructure is superior to creating a new extension.
  - **Context:** [#519: docs: document business-owned response value](https://github.com/Universal-Commerce-Protocol/ucp/pull/519)
- **Topic:** Base Action PR (PR #553)
  - **Notes:** Daniel Wyckoff presented the base action PR, proposing to move it forward independently of the ongoing 3DS discussions.
  - **Context:** [#553: feat: Introduce Actions and apply it to Checkout](https://github.com/Universal-Commerce-Protocol/ucp/pull/553)
- **Topic:** Returns Extension (PR #257)
  - **Notes:** Amit Handa raised concerns about generalizing too early, noting that warranties and subscriptions have very different costs and deductibles compared to standard returns. He also noted that return policies are typically universal, so using JSON paths for everything might be clunky. Vic Duggan suggested scoping the PR to pre-purchase return policies to reduce complexity.
  - **Context:** [#257: feat: add Return Extension to UCP specification](https://github.com/Universal-Commerce-Protocol/ucp/pull/257)
- **Topic:** Location Capabilities (PR #545)
  - **Notes:** Jing Li flagged the locations capability PR, which is related to the local fulfillment RFC, for an upcoming deep dive on Monday. It introduces capabilities to look up physical addresses and related details.
  - **Context:** [#545: feat: Introduce Location Search + Lookup capabilities](https://github.com/Universal-Commerce-Protocol/ucp/pull/545)
- **Topic:** WebAuthn PR (PR #483)
  - **Notes:** Amit Handa raised concerns about the WebAuthn PR relying on an unreleased IETF draft spec, which might change and lacks broad industry adoption. James Andersen and Vic Duggan discussed whether UCP should wait for greater web authentication adoption or proceed if it is compatible and grows in adoption.
  - **Context:** [#483: feat: Web Bot Auth (WBA) interop](https://github.com/Universal-Commerce-Protocol/ucp/pull/483)
- **Topic:** Additional Fields (PR #481)
  - **Notes:** Jing Li and Vic Duggan discussed PR 481 (Additional Fields), which allows for unstructured data blobs without creating heavy-duty extensions. This is a non-controversial feature that acts as a major unlock, especially for grocery vertical schemas.
  - **Context:** [#481: feat(shopping): Add additional fields proposal (from Shopify)](https://github.com/Universal-Commerce-Protocol/ucp/pull/481)
