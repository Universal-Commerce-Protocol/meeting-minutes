## 1. General Topics

- **Topic:** Protocol and Capability Versioning Strategy
  - **Notes:** The team debated the complexity of the current versioning structure, specifically whether to treat the protocol as an atomic bundle or separate the versioning of individual capabilities. Bundling everything together creates unnecessary version bumps for the entire protocol when only a single vertical undergoes changes, which confuses implementers and static analysis tooling. On the other hand, the current structure is confusing for integrators who may not know how to handle previous versions. TC discussed adopting a quarterly release train schedule to coordinate versions more effectively across verticals and agreed to revisit the discussion later.
  - **Context** [#499 RFC: Refactor of codebase to support capability versioning independent of protocol version](https://github.com/Universal-Commerce-Protocol/ucp/issues/499)
- **Topic:** Returns Extension
  - **Notes:** The team decided to sequence the Returns extension implementation after the Local Delivery/Fulfillment deep-dive.
  - **Context** [#257 feat: add Return Extension to UCP specification](https://github.com/Universal-Commerce-Protocol/ucp/pull/257)

## 2. Open Discussions & TC PR Reviews

- **Topic:** Service Advertisement Structure Strategy
  - **Discussion:** Discussed whether the protocol should explicitly define capability-specific service advertisements rather than relying on top-level service descriptions, which were described as somewhat "handwavy." Ilya Grigorik argued that the current two-stage negotiation process—where an integrator first verifies the domain and then traverses into the specific capability—is effective and that decomposing these structures would increase unnecessary complexity.
  - **Decision:** Aligned against unifying specific service advertisements at the top level. The team will instead establish a shared foundation for independent capabilities to carry their own service specifications, maintaining the current two-stage negotiation structure while establishing a normative convention or pattern for defining capability configurations.
  - **Context:** [#423 feat: delegated identity providers and accelerated IdP flow](https://github.com/Universal-Commerce-Protocol/ucp/pull/423)
- **Topic:** AP2 Signing Key Strategy
  - **Discussion:** Discussed technical alignment regarding the AP2 signing mechanism to enable cross-compatibility with web bot authentication.
  - **Decision:** The team aligned on the strategy to consolidate the signing mechanism to use 'keys' rather than 'signing keys' within the UCP schema to enable cross-compatibility (allowing a single set of keys to serve both AP2 and web bot).
  - **Context:** [#483 feat: Web Bot Auth (WBA) interop](https://github.com/Universal-Commerce-Protocol/ucp/pull/483)
