## 2. Technical Decisions & Consensus

- **Topic:** SDK Developer Experience & Dynamic Type Generation
  - **Notes:** Enric presented a proposal to generate typed SDKs from the specification schema via a continuous integration pipeline. Ilya Grigorik noted that while static type generation is valuable, the dynamic features of the protocol—including schema composition, runtime negotiation of custom extensions, and transport-level differences (REST vs. MCP)—require the SDK to handle dynamic runtime discovery, negotiation, and validation, rather than just static types.
- **Topic:** Capability Versioning and Operations
   - **Notes:** Amit Handa provided a heads-up that a PR is in progress to document capability versioning, resolving the operational impact of protocol-level releases on independent capability versions and their representation in public documentation.
- **Topic:** Repository Pre-commit Hooks Friction
  - **Notes:** Discussed local pre-commit and pre-push hook failures caused by newline validations on newly added assets (such as logos). Guillaume Verdier committed to submitting a PR to resolve this validation friction.
- **Topic:** Backlog Cleanup of Legacy Pull Requests
  - **Discussion:** Reviewed outstanding draft and superseded PRs to maintain repository hygiene. Proposal to close legacy loyalty PR #251 and superseded marketing consent PR #407.

## 3. Open Discussions & TC PR Reviews

- **Topic:** Buyer Consent
  - **Discussion:** Debated the V2 marketing consent model design. Ilya Grigorik advocated for a simplified structure where the business backend dynamically resolves appropriate consent defaults based on buyer jurisdiction, returning a single 'allowed' boolean bit. James Andersen and Amit Handa highlighted that UX checkbox presentation (default opt-in vs. opt-out) is distinct from actual user-confirmed consent, especially regarding regulatory disclosures. The group agreed that segment-specific settings should override global settings for symmetry, but postponed final alignment until Amit Handa posts request/response JSON examples. Evaluated ordered arrays versus map structures for specifying item priority (e.g., payment handlers). Map structures guarantee key uniqueness at the schema level, whereas arrays are prone to duplicate entries and validation issues.
  - **Context:** [#451 feat!: buyer consent with per-segment extensibility](https://github.com/Universal-Commerce-Protocol/ucp/pull/451)
- **Topic:** Signature and Profile Schema Interoperability
  - **Discussion:** Reviewed changes to align signature algorithms and profile schemas to prevent developers from needing to maintain redundant cryptographic keys and perform separate signing ceremonies.
  - **Context:** [#483 feat: Web Bot Auth (WBA) interop](https://github.com/Universal-Commerce-Protocol/ucp/pull/483)
- **Topic:** Recomputation of Idempotency Keys across Requests
  - **Discussion:** Reviewed clarification around the idempotency key contract to ensure consistency and prevent stale keys when request payloads change.
  - **Context:** [#485 fix: specify idempotency payload-mismatch contract](https://github.com/Universal-Commerce-Protocol/ucp/pull/485)
- **Topic:** Local Inventory & Fulfillment Roadmap Phasing
  - **Discussion:** Evaluated the local inventory RFC to prevent scope creep in catalog modeling, specifically regarding complex delivery options and pricing.
  - **Decision:** Aligned on a three-phase execution strategy for local product discovery: (1) Local Product Discovery (catalog lookup and store location status), (2) Buy & Collect Fulfillment (pickup/collection mechanics), and (3) Omni-channel complexities and split shipments.
  - **Context:** [#375 RFC Store-Based Local Inventory & Fulfillment Options](https://github.com/Universal-Commerce-Protocol/ucp/issues/375)
